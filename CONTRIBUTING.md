# Участие в разработке

## Git Flow

Проект использует упрощённую модель **Git Flow**.

### Ветки

| Ветка | Назначение |
|-------|-----------|
| `master` | Стабильный релизный код |
| `develop` | Интеграционная ветка — сюда сливаются все фичи |
| `feature/<название>` | Новая функциональность |
| `fix/<название>` | Исправление бага |
| `hotfix/<название>` | Срочное исправление в `master` |
| `release/<версия>` | Подготовка релиза (финальное тестирование, бамп версии) |

### Жизненный цикл ветки

```
master
  └── develop
        ├── feature/nlp-skill-gap    → PR → develop
        ├── feature/hh-pagination    → PR → develop
        └── fix/resume-upload-error  → PR → develop
                                           ↓
                                    release/1.1.0 → PR → master + тег v1.1.0
```

### Правила

- Никогда не коммитим напрямую в `master`.
- Ветку фичи создаём от `develop`, мержим обратно в `develop`.
- Хотфиксы создаются от `master` и мержатся в `master` **и** `develop`.
- Перед слиянием обязателен PR (Pull Request) и ревью хотя бы одного участника.

### Примеры команд

```bash
# Начало работы над фичей
git checkout develop
git pull origin develop
git checkout -b feature/add-filter-by-salary

# Публикация ветки
git push -u origin feature/add-filter-by-salary

# После апрувa PR — удалить локальную ветку
git branch -d feature/add-filter-by-salary
```

## Conventional Commits

Все коммиты должны соответствовать спецификации [Conventional Commits v1.0](https://www.conventionalcommits.org/).

### Формат

```
<type>(<scope>): <описание>

[опциональное тело]

[опциональный футер]
```

### Типы коммитов

| Тип | Когда использовать |
|-----|--------------------|
| `feat` | Новая функциональность |
| `fix` | Исправление бага |
| `docs` | Изменения только в документации |
| `style` | Форматирование, пробелы — без изменения логики |
| `refactor` | Рефакторинг без новой функциональности и без фикса |
| `test` | Добавление или исправление тестов |
| `chore` | Обновление зависимостей, конфигурации CI, инструментов |
| `perf` | Улучшение производительности |
| `ci` | Изменения в конфигурации CI/CD |

### Scope (область)

Опциональный контекст изменения в скобках: `bot`, `nlp`, `db`, `api`, `config`, `deps`.

### Примеры

```
feat(nlp): добавить определение уровня образования из резюме

fix(bot): исправить падение при отправке пустого файла резюме

docs: обновить README — добавить раздел про алгоритм ранжирования

refactor(db): вынести CRUD-операции в отдельный репозиторий

chore(deps): обновить aiogram до 3.7.0

feat(api)!: изменить формат ответа HH.ru — BREAKING CHANGE

BREAKING CHANGE: поле salary теперь возвращает объект вместо числа
```

### Breaking Changes

Ломающие изменения обозначаются `!` после типа или строкой `BREAKING CHANGE:` в футере:

```
feat(api)!: переработать схему БД для поддержки нескольких резюме
```

## Версионирование

Проект следует **Semantic Versioning** (`MAJOR.MINOR.PATCH`):

- `MAJOR` — ломающие изменения (`BREAKING CHANGE`)
- `MINOR` — новая функциональность (`feat`)
- `PATCH` — исправления (`fix`)

Теги создаются из ветки `release/*`:

```bash
git tag -a v1.1.0 -m "release: v1.1.0"
git push origin v1.1.0
```
